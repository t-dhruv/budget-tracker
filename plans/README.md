# Implementation Plans

Generated by the improve skill on 2026-07-06. These plans add admin UI tooling and automated database backup to the production Docker stack.

**Context**: The user requested database UI, Redis UI, and auto backup strategy for a personal budget tracking application (MoneyMatter). Recon showed:
- pgAdmin already exists in dev compose but not in prod
- No Redis admin tool exists anywhere
- CI-based backup exists (GitHub Actions → R2) but no local backup automation for self-hosted users

## Dependency notes

- All three plans are independent. Execute in any order.
- 003 is P1 because it addresses a safety gap (no local backups for self-hosters). 001 and 002 are P2 (developer convenience).

## Findings considered and rejected

- **Built-in in-app DB browser**: Too much effort (L) vs value. pgAdmin suffices and the project already has it in dev.
- **RedisInsight instead of Redis Commander**: Heavier container. Redis Commander is 5MB and has read-only mode. RedisInsight is better for deep debugging but overkill for queue inspection.
- **R2 upload from backup sidecar**: The CI backup already handles this for the maintainer's VPS. Adding it to the sidecar would duplicate functionality and require storing R2 credentials in more places. Future improvement if self-hosters request it.
- **Full backup service with encryption/PGP**: Over-engineering for a self-hosted personal finance app. Sidecar + CI backup covers the threat model (accidental deletion, corruption).
