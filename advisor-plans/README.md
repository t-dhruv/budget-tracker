# Implementation Plans — Category Taxonomy Improvements

Generated by the improve skill on 2026-07-08 for `main` at commit `5a6cea81`.

These plans expand and improve the default budget category taxonomy for a global (EU + NA) audience. The existing 11 main / 69 subcategory structure has gaps (no Health, Education, or Pets main categories) and naming overlaps that produce unreliable spending reports.

All three plans were cold-reviewed by subagents and revised on 2026-07-08. See individual plan files for the full correction log.

## Dependency notes

- **001 must execute before 002**: plan 002 populates the migration arrays that plan 001 creates. Without the migration framework, new categories only reach new signups.
- **002 must execute before 003**: plan 003 removes `shopping/pets-animals` from the seed, which only makes sense after plan 002 adds the `pets` main category. Without 002, users would lose the pets subcategory entirely.
- All three plans are additive from the data perspective: they never delete or modify existing users' category rows. Existing categorizations are preserved.

## Findings considered and rejected

- **Splitting "Life" into separate main categories beyond Health & Education**: The audit identified Fitness, Entertainment, and Hobbies as potential standalone categories. Rejected for this pass — the additive approach (add Health + Education, leave others under Life) has lower blast radius. Revisit if Life continues to grow as a junk-drawer.
- **Income/Expense type attribute on categories**: Making `income` vs `expense` a first-class category attribute would require model changes, CRUD changes, and budget logic changes. Worth exploring but scope is XL. Not included here.
- **Deleting old `shopping/pets-animals` rows from existing users**: Too destructive — would break existing transaction categorizations. Plan 003 only removes from the seed definition for new signups.
- **Country-specific seed sets (US vs UK vs DE vs FR)**: Vision-level idea. The current approach (one superset taxonomy for all) is simpler to maintain and avoids complexity in locale-to-taxonomy mapping. If analytics show users systematically hiding certain categories per region, revisit.
